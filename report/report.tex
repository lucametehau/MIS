\section{Introduction}
This report evaluates various implementations of the Maximal Independent Set (MIS) algorithm. MIS is a fundamental problem in graph theory with applications in distributed computing, wireless networks, and resource allocation. We compare a sequential implementation against parallel versions based on Luby's algorithm, including multi-core CPU and GPU implementations.

\section{Graph Representation}
The efficiency of graph algorithms is heavily influenced by how the graph is stored in memory. In this project, we implemented and compared two primary representations:

\subsection{Standard Adjacency List}
The \texttt{Graph} type is implemented as a \texttt{std::vector<std::vector<uint32\_t>>}. While this representation is intuitive and allows for easy graph construction, it has several drawbacks for high-performance computing:
\begin{itemize}
    \item \textbf{Memory Fragmentation}: Each vertex's adjacency list is a separate memory allocation, leading to scattered data across the heap.
    \item \textbf{Cache Misses}: Traversing the neighbors of multiple vertices often results in frequent cache misses due to the lack of data contiguity.
    \item \textbf{Overhead}: Each \texttt{std::vector} object carries internal metadata (pointers, size, capacity), which adds significant memory overhead for large graphs with millions of vertices.
\end{itemize}

\subsection{Compressed Sparse Row (CSR)}
To address the limitations of the adjacency list, we implemented the \texttt{GraphCSR} class. This format stores the entire graph structure using two contiguous arrays:
\begin{itemize}
    \item \textbf{\texttt{offsets}}: An array of size $N+1$ where \texttt{offsets[i]} marks the starting position of vertex $i$'s neighbors in the \texttt{edges} array.
    \item \textbf{\texttt{edges}}: A single array of size $M$ (total number of edges) containing all neighbor indices stored back-to-back.
\end{itemize}

The CSR format offers several advantages:
\begin{itemize}
    \item \textbf{Cache Locality}: Since all edges are stored contiguously, traversing neighbors is highly cache-efficient, significantly speeding up neighbor traversal phases.
    \item \textbf{Memory Efficiency}: CSR eliminates the metadata overhead of individual vectors and reduces memory fragmentation.
    \item \textbf{GPU Compatibility}: The contiguous nature of CSR makes it the standard format for graph processing on GPUs, as it allows for efficient coalesced memory accesses.
\end{itemize}

\section{Algorithms}
In this section, we describe the main algorithms implemented for the Maximal Independent Set (MIS) problem.

\subsection{Sequential MIS}
The sequential algorithm follows a greedy approach. It iterates through all vertices and adds a vertex to the MIS if none of its neighbors have been added yet. This algorithm has a linear time complexity of $O(N+M)$, where $N$ is the number of vertices and $M$ is the number of edges.

\begin{lstlisting}
for (std::size_t i = 0; i < n; i++) {
    if (active[i]) {
        mis.push_back(i);
        for (auto &son : g[i])
            active[son] = 0;
    }
}
\end{lstlisting}

\subsection{Luby's Algorithm \cite{luby1986}}
Luby's algorithm is a randomized parallel algorithm for finding a Maximal Independent Set. It is designed to work in a synchronous, round-based manner, making it highly suitable for parallel architectures like multi-core CPUs (via OpenMP) and GPUs (via CUDA).

The algorithm proceeds in discrete rounds, where each round consists of the following steps:
\begin{enumerate}
    \item \textbf{Priority Assignment}: Each currently active vertex $v$ independently chooses a random priority $p(v)$ from a large uniform distribution.
    \item \textbf{Local Maxima Selection}: A vertex $v$ is added to the independent set if its priority $p(v)$ is strictly greater than the priorities of all its active neighbors $N(v)$. Ties are typically broken using vertex indices.
    \item \textbf{Pruning}: Any vertex selected in the previous step, along with all of its neighbors, is marked as inactive and removed from further consideration.
\end{enumerate}

The process repeats on the remaining induced subgraph until no active vertices remain. 

\textbf{Complexity}: Theoretically, Luby's algorithm is very efficient. It has been proven that the algorithm finds a Maximal Independent Set in $O(\log n)$ rounds with high probability, where $n$ is the number of vertices. Since each round involves traversing edges, the total work is proportional to the number of edges $M$, but the "depth" of the computation is logarithmic, allowing for significant speedups on parallel hardware.

\subsubsection{OpenMP Parallelization}
We utilized OpenMP to parallelize Luby's algorithm on multi-core CPUs, given these factors:
\begin{itemize}
    \item \textbf{Shared Memory Model}: Luby's algorithm requires threads to frequently access the priorities and status of neighboring vertices. OpenMP's shared memory architecture allows all threads to access the global graph structure and priority arrays without the overhead of explicit message passing.
    \item \textbf{Data Parallelism}: The core operations of each round—priority assignment and local maxima selection—are naturally data-parallel. Using \texttt{\#pragma omp parallel for} allowed us to distribute these tasks across available CPU cores with minimal code modification.
    \item \textbf{Dynamic Load Balancing}: In irregular graphs, such as Scale-Free networks, vertex degrees vary significantly. OpenMP's \texttt{dynamic} scheduling helps mitigate the "straggler" problem where threads processing high-degree nodes would otherwise bottleneck the entire round.
    \item \textbf{Synchronization Primitives}: OpenMP provides efficient primitives like \texttt{atomic} and \texttt{critical} sections, which were essential for managing shared state updates (e.g., adding nodes to the MIS or updating the "any active" flag) safely across threads.
\end{itemize}

\subsection{Improved Luby's Algorithm}
The improved version refines the base parallel Luby algorithm by reducing synchronisation overhead,
eliminating explicit random number generation, and exploiting locality more effectively.

The key optimisations are:

\begin{itemize}
    \item \textbf{Active Node Tracking}:
        Instead of scanning the entire vertex set each round, a dedicated vector \texttt{active\_nodes}
        stores only those vertices still eligible for the independent set.
        After each round, survivors are compacted into a fresh list for the next iteration.

    \item \textbf{Deterministic, On‑the‑Fly Priorities}:
        Each vertex’s random priority is computed from the current round number and its own index
        using a fast, splitmix64 hash function.
        This avoids an expensive thread‑local RNG pass, eliminates the need for a global priority array,
        and guarantees a reproducible pseudo‑random order without additional memory traffic.

    \item \textbf{Unified Max‑Detection and Deactivation}:
        The search for local maxima and the immediate deactivation of winners and their neighbours
        are fused into a single parallel region.
        This fusion halves the number of global barriers per round compared to the basic version,
        reducing overhead.

    \item \textbf{Dynamic Load Balancing}:
        The neighbour‑checking loop uses \texttt{schedule(dynamic, 256)} to adapt to the
        highly irregular degree distributions of scale‑free and other non‑uniform graphs.

    \item \textbf{Thread‑Local Aggregation with Critical Merge}:
        Both the newly selected independent set vertices and the surviving active nodes
        are collected into thread‑local lists during their respective loops.
        A single critical section per thread then appends the local results to a global
        vector.
        This pattern is used in the unified max‑detection/deactivation region (for MIS
        additions) and again in the lightweight survivor compaction region.
\end{itemize}

\subsection{Luby GPU Implementation}
The GPU version of Luby's algorithm leverages the massive parallelism of CUDA architectures. The implementation is based on the CSR format and follows an iterative process where several kernels are launched per round:

\begin{itemize}
    \item \textbf{Iterative Kernel Execution}: The algorithm operates in rounds. In each round, we launch a sequence of specialized CUDA kernels: one to identify candidates (local maxima) and another to update the MIS and prune neighbors. This cycle continues iteratively until all vertices have been processed.
    \item \textbf{Fused Kernels}: We combine priority assignment and candidate selection into a single kernel (\texttt{select\_assign\_priorities\_fused\_candidates\_kernel}) to reduce global memory traffic and kernel launch overhead.
    \item \textbf{Restricted Pointers}: We utilize the \texttt{\_\_restrict\_\_} keyword for all input arrays (offsets, edges, active status). This informs the compiler that these pointers do not alias, enabling more aggressive load/store optimizations and better utilization of read-only data caches.
    \item \textbf{Deterministic Hashing}: Instead of generating and storing random numbers for all nodes in every round, we use a lightweight \texttt{hash\_priority} function based on the vertex ID and iteration number. This reduces memory bandwidth requirements as priorities are computed on-the-fly.
    \item \textbf{Minimized Host-Device Transfers}: Only a single integer (\texttt{any\_active}) is copied from device to host at the end of each round to determine convergence, keeping the hot loop mostly on the GPU.
\end{itemize}

\section{Experimental Setup}
To rigorously evaluate the performance of our implementations, we utilize two distinct random graph models that represent different topological characteristics commonly found in real-world applications.

\subsection{Erdős-Rényi Model (Uniform Sparse)}
The first model is the standard Erdős-Rényi random graph, specifically the $G(n, p)$ model~\cite{erdos1959}. In this model, each potential edge between any two vertices exists independently with a constant probability $p$. For our "Uniform Sparse" benchmarks, we set $p$ such that the average number of edges is $M = c \cdot N$, with $c$ typically set to $10$.

This graph type serves as a baseline for "average" case performance. The degree distribution follows a Poisson distribution for large $n$, meaning most vertices have approximately the same number of neighbors. This relative uniformity simplifies workload balancing across parallel threads.

\subsection{Barabási–Albert Model (Scale-Free)}
The second model is the Barabási–Albert (BA) model~\cite{barabasi1999}, which generates scale-free graphs using a preferential attachment mechanism. The process starts with a small clique of $m_0$ nodes, and new nodes are added one by one, each connecting to $m$ existing nodes with a probability proportional to their current degree.

Scale-free graphs are characterized by a power-law degree distribution, $P(k) \sim k^{-\gamma}$. These graphs contain "hubs"—a few vertices with extremely high degrees—while the vast majority of vertices have very few connections. This topology is representative of social networks and the World Wide Web. For MIS algorithms, scale-free graphs present a significant challenge for parallelization due to extreme workload imbalance, making optimizations like dynamic scheduling and CSR layout essential.

\section{Weighted MIS}

\subsection{Problem Formulation}

We recall that the standard Maximal Independent Set (MIS) is a subset of vertices $S\subset V$ of a graph $G=(V, E)$ such that no two vertices in $S$ are joined by an edge, and no new vertex can be added to $S$ without creating an edge. Now, we extend this framework by adding a weight $w_v\in \mathbb{R}^+$ for every vertex $v\in V$. In this setting, the goal is now to find a MIS with the maximum possible total weight \textbf{in practice}. It is important to note that finding the \textbf{maximum weight independent set} (i.e., the globally optimal solution) is NP-hard in general. In this view, we will focus on weight-based heuristic algorithms that find a valid MIS while favoring vertices with high weights, and then compare their solution quality empirically. 

\subsection{Algorithms}

We decided to study the performance of the following two algorithms, and then benchmark their performance based on the initial weight assignment. 

\subsubsection{Algorithm 1: Weighted Greedy}
A vertex $v$ is added to the MIS if and only if it has the highest weight among its active neighbors, with ties broken by vertex index:
\begin{equation}
  v \text{ wins} \iff \forall\, u \in N(v) \text{ active}:\quad w_v > w_u \;\lor\; (w_v = w_u \,\land\, v < u)
\end{equation}
This approach is based on Basagni~\cite{basagni2001}.
\subsubsection{Algorithm 2: Weighted Sampling}
Each active vertex $v$ draws $U_v \sim \text{Uniform}(0,1)$ and computes a priority:
\begin{equation}
  p_v = U_v^{1/w_v}
\end{equation}
Vertex $v$ wins the round if $p_v > p_u$ for all active neighbors $u$. That is, we have that the vertices with higher weights have the higher priority. 

\subsection{Weight Distributions}

\textbf{Uniform:} Each vertex $v$ is assigned a weight drawn uniformly at random:
\begin{equation}
  w_v \sim \text{Uniform}(1, 10)
\end{equation}
\textbf{Exponential:} Each vertex $v$ is assigned: (the $+1$ shift ensures $w_v > 0$, which is required by the power trick in Algorithm 2.)
\begin{equation}
  w_v = 1 + X, \quad X \sim \text{Exponential}(\lambda)
\end{equation}
\textbf{Clustered:} A fraction $\rho = 0.1$ of vertices are randomly designated as hotspots:
\begin{equation}
  w_v = \begin{cases} 10 & \text{with probability } \rho \\ 1 & \text{otherwise} \end{cases}
\end{equation}

\subsection{Parallelization}

Both weighted algorithms weree parallelized using OpenMP, following the same structure as the unweighted Luby. Each round consists of a parallel local-maxima selection phase (the sampling algorithm also does a separate parallel priority-assignment phase before that). Thread-local vectors collect candidates, which are merged via a \texttt{critical} section. The \texttt{any\_active} check uses an OpenMP reduction. All benchmarks in this section were run on the Polytechnique machines, namely Intel Core i7-13700K (16 cores, 24 threads). Thread counts of 1, 2, 3, and 4 were evaluated for both algorithms. The sequential baselines are the same algorithms without parallelization.

\subsection{Discussion}

We first look at (Table~\ref{tab:weighted_quality}) to investigate the solutuiob quality. The results indicate that the Greedy approach consistently finds the MIS with higher total weight across all graph types and initial weight distributions. This can be explained by the fact that stochastic sampling can occasionally make the algorithm give a low-weight neighboring vertex a higher priority, thus permanently excluding the high-weight vertex from the MIS selection.
We also see that on Scale-Free graph types, the mean weight results are much closer than those on the Uniform-Sparse graph types (gap of 11--20\% on Scale-Free vs.\ 21--38\% on Uniform Sparse).
We see that the exponential and clustered initial weight distributions improve greedy's advantage (38\% and 31\% gaps respectively on Uniform Sparse, vs.\ 21\% for uniform weights). The reason for this is that in these settings, we have relatively fewer very high-weight vertices, and the greedy approach always captures them. Meanwhile, the sampling approach may exclude them in case a low-weight vertex is selected instead at random.


As for parallelization,  (Table~\ref{tab:weighted_thread_scaling}), using 1 thread is slightly slower than the sequential baseline due to OpenMP overhead. We can see that both algorithms consistently scale as we increase the number of threds, both achieving roughly a 2x speedup at 4 threads. We can see that the speedup is sublinear, however, this is expected as because in the code each rounds inehrently has sequential parts. For instance, he critical-section merge of candidate vertices, and the serial deactivation of selected vertices and their neighbors.

Finally, (Table~\ref{tab:weighted_density}) describes the density variation results. We can see that as we scale $c$, the absolute time increases for both algorithms, which is expected. However, the speedup improvecs with density from 2.1x to 2.9x as $c$ grows, since denser graphs give each thread more work to do, so the fixed cost of managing threads matters less.

\section{Conclusion}
In this project, we evaluated multiple implementations of the Maximal Independent Set (MIS) algorithm, spanning sequential, multi-core CPU, and GPU architectures. Our results demonstrate that the choice of the optimal algorithm depends heavily on the graph topology, problem size, and hardware availability.

Key findings include:
\begin{itemize}
    \item \textbf{Data Representation}: The Compressed Sparse Row (CSR) format is foundational for high-performance graph processing. It consistently outperformed the standard adjacency list by improving cache locality and reducing memory overhead, providing a significant speedup even for sequential execution.
    \item \textbf{Parallel Efficiency}: While the base Luby's algorithm provides a theoretical parallel framework, its naive implementation suffers from significant overhead. Our \textit{Improved Luby} implementation, utilizing deterministic on-the-fly priorities and active node tracking, achieved substantial speedups on multi-core CPUs, particularly on large-scale irregular graphs where it often outperformed the GPU version due to better handling of workload imbalance.
    \item \textbf{GPU Acceleration}: The CUDA-based implementation excels on large, dense graphs where the massive parallelism of the GPU can be fully leveraged. However, for smaller graphs or highly irregular scale-free networks, the overhead of kernel launches and memory latency can make CPU-based parallelization more competitive.
    \item \textbf{Weighted MIS}: In the weighted variant, the Greedy approach demonstrated superior solution quality across all tested weight distributions. While randomized sampling offers a different trade-off, the greedy heuristic's ability to prioritize high-weight nodes makes it the preferred choice for applications where total weight is the primary metric. Both weighted algorithms also achieved roughly 2× speedup over their sequential baselines at four threads.
\end{itemize}

Overall, the transition from sequential to optimized parallel implementations provides orders of magnitude improvement in performance, enabling the processing of graphs with tens of millions of vertices in under a second.
